# 劍網三吃雞影片數據分析工具 - 完整需求規格書 (無 API 額度限制離線版)

## 前言與專案背景

本文件為《劍網三》絕境戰場（吃雞模式）影片數據分析工具之完整需求規格書 (Software Requirements Specification, SRS)。

隨著遊戲實況生態的蓬勃發展與 VOD（隨選視訊）數量的日益龐大，高端玩家與賽事數據分析師對於「對局覆盤」與「大數據統計」的需求急遽增加。然而，傳統仰賴人工觀看影片、手動記錄縮圈時間與馬匪刷新點的方式，不僅極度耗時、容易產生人為疏漏，更完全無法應對動輒數百小時的影音資料量，這種土法煉鋼的模式在現代數據分析中已不切實際。

本規格書應運而生，旨在提供一套具備高度實作可行性、極致效能壓榨與嚴謹防呆設計的自動化開發藍圖。文件中不但詳細定義了基於 GitHub Actions 的高併發分散式技術架構、標準化的關聯式資料庫輸出模型，更針對遊戲複雜的 UI 特性，量身打造了專屬的影像預處理 (Image Preprocessing) 與光學字元辨識 (OCR) 邏輯。

此外，考慮到實況影片的不可控性，本規格書深度涵蓋了諸如「實況主中途錄影」、「玩家提早重傷退場」、「伺服器斷線重連」等各種極端邊界情況 (Edge Cases) 的狀態機異常處理機制。我們的核心目標，是徹底消弭不同開發者、工程師之間的認知落差。確保任何接手此專案的軟體開發者，只要嚴格依循此文件中的各項規範與技術指標，皆能毫無歧義地開發出功能表現、容錯能力與最終報表結構完全一致，且堅若磐石的自動化分析系統。

## 1. 專案目標與系統架構

本系統的最高指導原則為建構一套完全免費、無第三方 API 額度限制（如 OpenAI 或 Google Vision API），且不消耗使用者本機端運算資源的離線影像辨識與數據分析系統。系統將全自動化地從指定的 YouTube 頻道、自訂播放清單或單一影片中，精準解析出縮圈時間軸以及馬匪刷新時的各項戰略指標。

### 1.1 核心技術堆疊與選型考量

- **自動化執行平台 (CI/CD)：** GitHub Actions (GHA) Ubuntu 虛擬機
  - **選型考量：** 充分利用 GitHub 針對開源/公開儲存庫提供的免費額度，作為背景執行的算力池。開發者或使用者無需自行架設伺服器，真正實現零成本維運。

- **並行運算架構 (Massive Parallelism)：** GHA Matrix Strategy (叢集多開) + Python `multiprocessing` (單機多核 Worker)
  - **選型考量：** 影像處理屬於高度 CPU 密集型任務。透過 Map-Reduce 模式，將數百支影片分發給多台虛擬機（叢集），並在每台機器內部榨乾多核心算力，可將原本需要數週的分析時間壓縮至幾小時內完成。

- **影片串流處理 (Streaming & 閱後即焚)：** `yt-dlp` + OpenCV (需確保帶有 FFmpeg 後端)。
  - **強制規範與資安考量：** 絕對禁止將影片完整下載至實體硬碟 (Disk I/O)。動輒幾十 GB 的高畫質 VOD 會瞬間塞爆 GHA 的磁碟空間。程式須透過 `yt-dlp` 擷取 HLS/m3u8 串流直連網址，並設定 OpenCV 透過網路直接將影格載入記憶體 (RAM) 中運算。分析完畢的影格必須立刻觸發垃圾回收 (Garbage Collection) 釋放資源，絕對不儲存、不保留、不備份任何影片實體檔案或截圖。

- **影像處理演算法 (Computer Vision)：** OpenCV (`opencv-python-headless`)
  - 負責影像的降噪、統一解析度映射、灰階轉換、自適應二值化 (Adaptive Thresholding)、形態學處理 (Morphological Transformations) 以及精準的 ROI (Region of Interest) 裁切。

- **核心文字辨識引擎 (OCR)：** Tesseract-OCR 與 `pytesseract`。
  - **語言包限制：** 因遊戲介面 UI 為純簡體中文環境，為極大化辨識準確率並縮短推論時間，僅需且僅能安裝 `tesseract-ocr-chi-sim` 與 `tesseract-ocr-eng`。嚴禁掛載 `chi_tra` (繁體中文) 語言包，以避免 Tesseract 在特徵提取時因簡繁字體過於相近而產生無謂的誤判（例如將簡體的「门」誤認為其他符號）。

- **數據整合與輸出 (Data Processing)：** Pandas 與 `openpyxl`。

## 2. 輸入與影片來源解析需求

為達到完全自動化的批次處理，系統必須具備強大的網址解析與排重能力。

### 2.1 影片連結輸入點 (Input Source)

系統唯一認可的輸入源為專案根目錄下的 `videos.txt` 檔案。

- **檔案格式規範：** 每行填寫一個標準 HTTP/HTTPS 網址，系統必須具備過濾空白行與註解的功能（支援以 `#` 開頭作為註解忽略標記）。
- **範例結構：**
  ```text
  # 這是 2024 年大師賽的 VOD 清單
  https://www.youtube.com/watch?v=example1
  # 下方是知名實況主的整個吃雞精華播放清單
  https://www.youtube.com/playlist?list=PL_example2
  ```

### 2.2 頻道與播放清單動態展開 (Channel & Playlist Expansion)

- **功能需求：** 當使用者填入的是 YouTube 頻道首頁或是包含多支影片的播放清單網址時，系統必須能將其自動「解壓縮」展開為單一影片連結的扁平化清單 (Flat List)。
- **開發實作細節：**
  - 使用 Python 呼叫 `yt-dlp` 時，必須帶入 `extract_flat=True` 參數。此參數能讓工具僅爬取中繼資料 (Metadata) 而不實際解析影片，極大幅度提升抓取速度。
  - 從返回的 JSON 結構中提取 `entries` 陣列裡的 `id` 欄位，自動重組為標準格式 `https://www.youtube.com/watch?v={id}`。
  - **關鍵過濾機制：** 必須檢查中繼資料，嚴格過濾掉狀態為正在直播中 (`is_live=True`) 或即將首播的影片，僅保留已經封裝完成的 VOD 存檔，避免串流讀取失敗。

### 2.3 增量辨識與中斷恢復 (Incremental Analysis & Resume)

- **功能需求：** 為了節省極其寶貴的運算資源，系統必須具備「記憶」能力，絕對避免對已經成功分析過的影片進行重複運算。
- **開發實作細節：**
  - 在啟動任何 OpenCV 讀取作業前，程式必須先掃描並嘗試讀取當前儲存庫中的 `analysis_results.xlsx` 檔案。
  - 解析分頁「對局主表 (Match_Master)」中的「影片連結 (Video_URL)」欄位，建立一個全域的 HashSet (雜湊集合) 快取。
  - 當系統準備處理 `videos.txt` 中的網址時，先進行 O(1) 的雜湊比對。若該網址已存在於快取中，則在 Log 中印出 `[Skip] Already Analyzed: <URL>`，並直接全域跳過該部影片的所有後續處理流程。

## 3. 影格抽取與預處理管線 (Image Processing Pipeline)

這是決定整個系統效能與 OCR 辨識成功率的最核心模組，必須嚴格依照以下順序實作。

### 3.1 抽樣頻率 (Sampling Rate & Performance Tuning)

- **硬性規定：** 系統強制設定採樣頻率為 0.5 FPS（即每 2 秒抽取 1 幀畫面）。
- **實作原由：** 逐幀分析 (30 或 60 FPS) 會讓系統陷入運算泥沼，處理一部 2 小時的影片需要耗費數十小時，毫無意義。吃雞模式中的縮圈與馬匪公告，其 UI 停留時間皆超過 3 秒，因此 0.5 FPS 既不會遺漏任何關鍵事件，又能將運算量縮減至原來的 1/120。
- **開發手段：** 開發者嚴禁使用無腦的 `cap.read()` 搭配計數器空轉（這依然會消耗解碼資源）。必須使用 OpenCV 的 `cv2.CAP_PROP_POS_MSEC` 屬性，直接精準跳躍 (Seek) 到指定的時間戳記提取影格，以避開不必要的影片解碼效能浪費。

### 3.2 畫面標準化與動態校正 (Resolution Normalization - Critical)

為了解決實況主上傳影片時畫質不一（如 720p, 1080p, 2K, 4K），導致固定座標裁切完全錯位崩潰的致命問題，影像裁切前必須執行以下標準化步驟：

- **統一解析度映射 (Normalization)：** 無論來源影片解析度為何，抽出的原始影格第一步必須強制作 `cv2.resize` 縮放至 1920x1080 解析度。此舉能確保後續所有的坐標參數皆建立在同一個幾何基準面上。
- **設定檔讀取 (`config.json`)：** UI 的坐標配置必須獨立抽離至 JSON 檔案中。支援採用絕對像素值 (基於 1920x1080 基準) 或 相對比例值 (0.0~1.0) 儲存。
  ```json
  {
    "ui_anchors": {
      "system_time": { "y_range": [0.0, 0.05], "x_range": [0.08, 0.25] },
      "system_chat": { "y_range": [0.20, 0.65], "x_range": [0.0, 0.25] },
      "circle_info": { "y_range": [0.55, 0.68], "x_range": [0.70, 0.85] },
      "players_alive": { "y_range": [0.20, 0.35], "x_range": [0.90, 0.98] }
    },
    "padding": 15
  }
  ```
- **UI 版面位移防呆保護 (Padding/Margin)：** 實況影片常因黑邊、OBS 擷取誤差造成 UI 輕微位移。程式在進行 ROI 裁剪時，必須自動將 `config.json` 內設定的寬高矩陣，向外擴張 15 像素 (Padding) 作為安全保護區。

### 3.3 基準坐標範圍配置 (基於 1920x1080 預估)

以下提供各關鍵 UI 的參考位置與 OCR 引擎配置參數，開發者可據此製作校正工具：

| 資訊區塊 | 預估 Y 軸 (Top-Bottom) | 預估 X 軸 (Left-Right) | OCR 語系參數 (`--oem 3 --psm`) | 區塊特徵說明 |
| :--- | :--- | :--- | :--- | :--- |
| **系統時間** | 0% ~ 5% | 8% ~ 25% | `eng` (psm 7) | 位於畫面極左上方，僅包含數字與冒號。 |
| **系統頻道監測** | 20% ~ 65% | 0% ~ 25% | `chi_sim` (psm 6) | 位於左下方，負責輸出廣播公告，為滾動文字區。 |
| **圈數資訊** | 55% ~ 68% | 70% ~ 85% | `eng` (psm 7) | 位於右下角小地圖上方，顯示如 `1/8` 格式。 |
| **剩餘人數** | 20% ~ 35% | 90% ~ 98% | `eng` (psm 7, 白名單 0-9) | 位於右側偏上，需透過白名單設定排除非數字干擾。 |

### 3.4 圖像優化與降噪演算法 (Image Enhancement)

裁剪出的 ROI 區塊影像非常小且模糊，必須依序通過以下過濾管線，才能餵給 Tesseract，藉此將辨識率從 40% 提升至 95% 以上：

- **放大 (Upscaling)：** 將 ROI 區塊長寬各放大 2~3 倍，插值法必須使用 `cv2.INTER_CUBIC`，確保字體邊緣平滑而不產生鋸齒。
- **灰度化 (Grayscale)：** 摒棄色彩干擾，使用 `cv2.cvtColor(img, cv2.COLOR_BGR2GRAY)` 轉換為單通道。
- **對比增強 (Contrast Normalization)：** 遊戲 UI 常有半透明底色，需套用 CLAHE (對比度限制自適應直方圖均衡化) 或簡單的線性對比度拉伸，將文字與背景的差異極大化。
- **二值化 (Thresholding)：** 最後使用 `cv2.THRESH_BINARY_INV` + `cv2.THRESH_OTSU`，讓演算法自動尋找最佳閾值，將影像徹底轉換為純黑底白字（或純白底黑字，需符合 Tesseract 的辨識偏好）。

## 4. 數據辨識邏輯與複雜狀態機架構 (State Machine)

此模組是整個分析工具的「大腦」，負責從 OCR 產出的混亂文字中萃取邏輯，並精準判斷遊戲的進程。

### 4.1 縮圈時間判定與 OCR 容錯機制 (Fault Tolerance)

- **文字清洗 (String Sanitization)：** 經過影片壓縮，Tesseract 極易產生視覺混淆。程式必須強制實作字元替換邏輯：
  - 圓弧形誤判：將 `O`, `o`, `D` 強制替換為 `0`。
  - 直線形誤判：將 `l`, `I`, `1`, `|` 強制替換為 `/` (針對圈數中間的斜線)。
  - 複雜形誤判：將 `B`, `&` 強制替換為 `8`。
- **正則特徵提取：** 文字清洗完畢後，嚴格匹配 Regex `([0-8])\s*/\s*8`，捕捉當前圈數。
- **判定邏輯 (狀態改變觸發 - Transition Trigger)：** 這是一項鐵律。系統必須觀測到數值的動態變化（例如從 N 變成 N+1 的那一瞬間），才能去記錄該變化首次出現時的「系統時間」。若影片一開局直接讀取到某個數字（例如 3/8），該數字僅能作為系統內部的「基準水位線」，絕對不予記錄該圈時間。
- **防雜訊與新局衝突 (Critical)：** 若數值突然變小（如 4/8 突然跳回 0/8），不可草率視為新對局。為了防止單一影格因特效遮蔽導致的 OCR 辨識崩潰，必須連續 3 個抽樣幀（約 6 秒鐘）皆穩定辨識出變小後的數值，才合法觸發 [4.3] 的「圈數遞減強制重啟」邏輯。若未滿 3 幀即恢復原高數值，則視該次突變為 OCR 雜訊，直接丟棄。

### 4.2 馬匪刷新判定 (Bandit Spawn Detection)

- **字串融合：** 移除系統頻道 ROI OCR 結果中的所有換行符號 `\n` 與空白字元，將整段頻道的對話融合成一條巨型字串。
- **關鍵字偵測：** 使用正則表達式在巨型字串中尋找 `前往`、`速往`、`出现了` 等簡體中文專屬的馬匪廣播特徵詞。
- **時間戳溯源 (Timestamp Traceback)：** 一旦偵測到關鍵字命中，程式必須以該關鍵字為起點，利用 Regex `\[\d{2}:\d{2}:\d{2}\]` 在該巨型字串中「向左尋找」最鄰近的時間戳記，此時間戳即為最精準的系統發布時間。

### 4.3 對局情境判定狀態機 (Match State Machine - 核心亮點)

一部數小時的影片通常包含數十場對局，且交雜著排隊大廳、商城瀏覽等無關畫面。開發者必須實作以下嚴謹的狀態機 (State Machine) 邏輯，來精確切割並封裝每一場對局的生命週期：

- **無效影片過濾 (No Match Detected)：** 若整部影片掃描完畢，從未成功辨識出任何有效的「X/8 圈數」，代表該 VOD 可能是競技場、副本或純粹聊天，未進入吃雞地圖。直接捨棄，不寫入任何資料至 Excel。
- **對局開始 / 新對局判定 (進入 State: `IN_MATCH`)：** 滿足以下任一條件時，狀態機即視為一場全新對局展開，立即產生一組獨一無二的 `Match_ID`，並重置圈數與時間計數器：
  - **完美開局 (精準第 0 圈起點)：** 偵測到系統頻道出現 `开启倒计时`，隨後偵測到 `计时:0`。這是遊戲最正規的起始點，直接記錄畫面左上角當下時間，作為第 0 圈 (`Circle_Level = 0`) 的官方開始時間。
  - **圈數開局 (未捕捉到倒數計時)：** 連續 3 幀偵測到圈數為 0/8。此情況判定已進入備戰，但不記錄 0 圈時間（因錯失了確切開局點），將此 0/8 設為狀態機基準點，靜待跳轉至 1/8 才開始紀錄。
  - **圈數遞減強制重啟 (重新排隊)：** 依據 [4.1] 邏輯，若當前有效圈數連續 3 幀小於歷史最高圈數，代表玩家在前一局死亡後迅速排進了新的一局。系統必須立即截斷舊對局，並初始化新對局。
- **中途錄影/加入 (Mid-Match Start)：** 若狀態機處於 `OUT_OF_MATCH` 待命狀態（例如影片剛開始），但連續 3 幀偵測到大於 0/8 的進階圈數（例如一開始就是 3/8）。此情況下，系統接受已身處對局中，但為了保證數據公信力，錯失的歷史圈數（0~2圈）與當前起始圈數（第 3 圈）皆不記錄時間。
- **對局終止 (進入 State: `OUT_OF_MATCH`)：** 當程式連續 10 個抽樣影格 (約 20 秒長度) 完全無法從右下角辨識出任何符合「X/8 圈數」或「剩餘人數」特徵時，或者遇到影片實體播畢 (EOF)，狀態機即判定該局結束（可能為玩家提早重傷退回大廳，或實況中斷）。將狀態優雅切換回備戰，將已收集的數據封裝寫入，並等待下一次的新局條件觸發。

## 5. 輸出數據規範與報表格式 (Data Schema & Mock Data)

系統分析完畢後，唯一的最終產出物為單一 `analysis_results.xlsx`。為了讓後續的大數據分析（如導入 Tableau、PowerBI 或 SQL 資料庫）更為順暢，資料結構必須符合嚴格的關聯式資料庫 (Relational Database) 原則，以 `Match_ID` 作為關聯主鍵 (Primary Key)。

### 分頁 1：對局主表 (Match_Master)

紀錄每場對局的巨觀資訊。

## 6. 大規模並行部署與執行規範 (GitHub Actions Architecture)

為了應對幾百甚至上千小時的影片解析，開發者必須採用以下 Map-Reduce 兩階段雲端架構來實作 CI/CD 工作流程 (Workflow)。此架構是為了解決「多台虛擬機同時分析完畢，同時向同一個 Excel 檔案執行 Git Push 造成嚴重 Merge Conflict (檔案衝突)」的災難性問題。

### 第一階段：分散運算叢集 (Map Phase - Matrix Strategy)

- **工作清單自動分割 (Job Sharding)：** 建立一個前置腳本 `split_jobs.py`，讀取 `videos.txt` 後，將百條網址清單均分為 N 個小批次 (Batch)，並將分割結果以 JSON Array 的形式輸出給 GHA 環境變數。
- **動態矩陣觸發 (GHA Matrix Expansion)：** 利用 GHA 的 `strategy: matrix` 語法，讀取上述的 JSON 陣列，動態且平行地啟動 N 台 Ubuntu 虛擬機（例如同時啟動 20 台機器，每台負責 5 支影片）。
- **單機多核壓榨 (CPU Multiprocessing)：** 在每一台虛擬機運行的 Python 腳本中，使用 `concurrent.futures.ProcessPoolExecutor` (Worker 數量建議設為系統 CPU 核心數 - 1)，並發處理分配到的 5 支影片，達到運算資源 100% 滿載。
- **防超時與網路重試 (Resilience)：** 由於 `yt-dlp` 擷取 YouTube 串流可能因網路波動或簽名過期而 Timeout，腳本內需包裝實作帶有指數退避 (Exponential Backoff) 的最多 3 次 Retry 機制。
- **純淨暫存輸出 (Upload Artifacts - 重中之重)：** 嚴禁在第一階段的任何虛擬機中執行 `git commit` 或 `git push`！ 分析完畢後，僅允許將該機器負責產出的部分資料存為碎片化的 CSV 表格 (`chunk_1.csv`, `chunk_2.csv`)，並透過 GitHub 的 `actions/upload-artifact` 套件上傳至雲端暫存區保管。

### 第二階段：資料聚合與自動提交 (Reduce Phase - Merge Job)

- **依賴設定與執行條件 (Job Dependencies & Conditions)：** 建立一個專屬的合併 Job。必須在 yaml 中明確設定 `needs: [matrix_jobs]`，這是 Map-Reduce 架構確保順序正確的絕對核心。為最大化資料收集率，應設定 `if: always()`，確保即使第一階段有極少數機器因崩潰失敗，系統仍能繼續收集其他 19 台機器產出的健康資料碎片。
- **資料下載、合併與深度去重 (Data Aggregation & Deduplication)：** 使用 `actions/download-artifact` 抓取所有 CSV 碎片檔案。利用 Python Pandas 讀取所有碎片，並以 `pd.concat()` 進行 DataFrame 實體合併。合併後，必須嚴格針對 `Match_ID` 執行去重 (`drop_duplicates()`)，防止同頻道重疊影片遭不同機器重複解析。接著，根據 `Video_URL` 與 `Game_Start_Time` 進行遞增排序 (Sort)。
- **報表格式化工程 (Final Excel Formatting)：** 利用 Pandas 搭配 `openpyxl` 引擎，將乾淨的 DataFrame 匯出至最終的 `analysis_results.xlsx`，並切分為三個獨立分頁。寫入時，應透過程式碼自動加入「凍結首列 (Freeze Panes)」與「根據內容自動調整欄寬 (Auto-fit Columns)」的優化，讓資料分析師打開檔案時無需二次排版。
- **自動化 Git 提交與防呆機制 (Automated Git Push & Guardrails)：** 由這台唯一的合併虛擬機全權負責更新 Repository，徹底杜絕併發衝突。執行 `git commit` 前，必須透過指令 (如 `git status --porcelain`) 檢查檔案是否有實質性的 Hash 變動。若無新增數據，腳本必須優雅退出 (Graceful Exit, `exit 0`)，避免觸發 "nothing to commit" 錯誤導致 Action 亮紅燈報錯；若有變動，則順利執行 `git commit` 與 `git push`，完美閉環全自動化的資料更新流程。
