# 劍網三吃雞影片數據分析工具 - 完整需求規格書 (無 API 額度限制離線版)

本文件為《劍網三》絕境戰場（吃雞）影片數據分析工具之完整需求規格書。旨在提供清晰的技術架構、輸入與輸出規範、影像辨識 (OCR) 邏輯，確保任何開發人員依循此文件皆能開發出功能完全相同的系統。

---

## 1. 專案目標與系統架構
本系統定位為**完全免費、無 API 額度限制且不消耗本機資源**的離線影像辨識系統。系統會自動從 YouTube 頻道、播放清單或單一影片中，解析出縮圈時間以及馬匪刷新時的各項對局指標。

### 1.1 核心技術堆疊
* **自動化執行平台**：GitHub Actions (GHA) Ubuntu 虛擬機（提供免費背景執行環境，不佔本機 CPU/頻寬）。
* **影片串流與下載**：`yt-dlp` (負責解析 YouTube 連結，包含頻道所有 Streams 展開)。
* **影像處理**：`OpenCV (opencv-python)` (影格讀取、圖像放大、灰階、二值化、ROI 裁切)。
* **文字辨識**：`Tesseract-OCR` 與 `pytesseract` 簡體中文語言包 (`tesseract-ocr-chi-sim`)。
* **數據分析與匯出**：`Pandas` 與 `openpyxl` (資料去重、狀態機判斷、匯出至多分頁 Excel)。

## 2. 輸入與影片來源解析需求

### 2.1 影片連結輸入點
* 系統需讀取一個位於專案根目錄的 `videos.txt` 檔案。
* 檔案格式：每一行代表一個輸入連結，支援以 `#` 開頭作為註解忽略。

### 2.2 頻道與播放清單展開 (Channel Expansion)
* **需求**：當使用者在 `videos.txt` 中填入頻道串流網址（如 `https://www.youtube.com/@cit5055/streams`）時，系統必須能自動展開該頻道下所有的影片連結。
* **開發細節**：
  * 使用 `yt-dlp` 的 `extract_flat=True` 模式，傳入頻道網址。
  * 提取 entries 中的影片 `id`，自動重組為標準的單一影片連結 `https://www.youtube.com/watch?v={id}`，將其展平成單一影片網址列表。

### 2.3 增量辨識 (Incremental Analysis)
* **需求**：為避免重複下載與辨識已完成的影片，系統在處理每一部影片前，必須檢查產出的 Excel 檔案。
* **開發細節**：
  * 若報表 `analysis_results.xlsx` 已存在，讀取「縮圈時間統計」分頁中已有的「影片連結」清單。
  * 若當前影片網址已在清單中，直接跳過 (Skip) 該影片，繼續處理下一個。這對處理擁有 180+ 影片的頻道尤為關鍵。

## 3. 影格抽取與 ROI (感興趣區域) 坐標配置需求

### 3.1 動態校正與設定檔機制 (重要開發要求)
> [!IMPORTANT]
> **開發核心細節**：由於本專案影片源自於不同帳號與電腦錄製（有些可能是全螢幕、視窗化、或經過局部裁剪的影片），且遊戲內的 UI 縮放比例（UI Scale）與排版均可能有些許公分的差異。
>
> 因此，系統**絕對禁止硬編碼 (Hardcode) 任何固定裁切坐標**。開發者**必須**實作以下動態校正配置機制：
> 
> 1. **設定檔讀取 (`config.json`)**：
>    * 程式必須從外部 JSON 設定檔讀取四個關鍵辨識區域 (ROI) 的範圍坐標。
>    * 坐標應採用**相對比例值 (Normalized Coordinates, 範圍 0.0 ~ 1.0)** 儲存，以便自動適配 1080p、720p 或其他視訊解析度。
> 2. **單幀校正模式 (Calibration Mode)**：
>    * 開發者應提供一輔助腳本（例如 `calibrate_regions.py`），該腳本可讀取目標頻道影片的一張完整截圖，並允許使用者以拖曳（或輸入數值）方式，標定出該影片源的「系統時間」、「系統頻道監測」、「圈數資訊」、「剩餘人數」四個紅橘框範圍，並將標定後的坐標自動寫入 `config.json`。

### 3.2 基準坐標範圍 (供校正參考之預估比例)
以下為標準 1080p 全螢幕《劍網三》播放畫面下（已與您提供的全螢幕截圖校對對齊）的**預估基準比例**（實際執行時會因錄製來源位移數公分，需讀取 `config.json` 校正值）：

| 資訊區塊 | 畫面位置說明 | 預估 Y 軸比例 (高度) | 預估 X 軸比例 (寬度) | OCR 語系限制 |
| :--- | :--- | :---: | :---: | :---: |
| **系統時間** | 畫面最左上角之時間欄 | `0.0%` ~ `3.5%` | `9.0%` ~ `22.0%` | 數字與 `:` |
| **系統頻道監測**| 左下角系統頻道滾動區 | `20.0%` ~ `60.0%` | `0.5%` ~ `25.0%` | 中文與數字 |
| **圈數資訊** | 右下角小地圖上方區塊 | `55.0%` ~ `66.0%` | `70.0%` ~ `84.0%` | 中文與數字 |
| **剩餘人數** | 右中偏上紅字「剩餘人數」及數字區 | `22.0%` ~ `35.0%` | `90.0%` ~ `97.0%` | 僅數字 |


### 3.3 UI 版面位移防呆機制 (UI Shift Mitigation)
* **實作方式**：
  * 開發者從 `config.json` 取得使用者校正後的 ROI 坐標後，程式裁剪時**必須自動將該區域的寬度與高度外擴 $10\% \sim 15\%$** 作為安全保護區。
  * 雖然外擴裁切會辨識出一些無關文字，但開發時必須使用**正規表示式 (Regex)** 與**關鍵字過濾**將其剔除。例如：在右下角大範圍 OCR 中，僅搜尋符合 `\d/8` 格式的字串；在右側剩餘人數大範圍中，僅抓取緊接在「剩余人数」下方的數字。
  * *優勢*：確保當影片 UI 因不同帳號有些許公分（或像素）的微幅位移時，仍能被安全保護區完整覆蓋，保證辨識不中斷。

## 4. 圖像預處理演算法

由於遊戲畫面背景複雜（有沙地、綠地、人物等背景干擾），直接進行 OCR 的精準度極低。裁剪出的 ROI 區塊**必須**依序進行以下預處理：

1. **影像放大 (Upscaling)**：將 ROI 影像長寬各放大 2 倍（使用雙立方插值 `cv2.INTER_CUBIC`），將字體線條加粗，利於小字辨識。
2. **灰度化 (Grayscale)**：去除色彩干擾，轉為單通道灰度圖。
3. **二值化 (Thresholding)**：使用 Otsu 二值化演算法（`cv2.THRESH_BINARY + cv2.THRESH_OTSU`），自動計算閾值並將圖像轉為純黑白（白底黑字或黑底白字），以徹底消除遊戲背景雜訊。

## 5. 數據辨識與狀態機邏輯

### 5.1 縮圈時間判定 (表格一)
* **OCR 偵測文字**：`已刷圈/总圈数: X/8`。
* **判定邏輯**：
  * 當 X 的數值增加時（如 `-1` 轉變為 `0`，`0` 轉變為 `1` ... 直至 `8`），記錄該變化首次出現時的「系統時間」。
  * 系統時間取得優先序：優先使用系統頻道監測中該影格最新訊息的 `[HH:MM:SS]`，若無則使用左上角系統時間辨識結果。
* **防誤判機制**：因為 OCR 偶爾會因字體模糊出現雜訊，若偵測到的 X 數值發生「倒退」或「突變」（如從 `1` 突變為 `7`），應予以忽略，只有遞增的變化才視為有效縮圈。

### 5.2 馬匪刷新判定 (表格二)
* **OCR 偵測文字**：在系統頻道監測區域中，尋找關鍵字 **「前往」** 或 **「速往」**。

#### 5.2.1 系統頻道換行與斷詞防呆處理 (Text Wrap Handling)
> [!WARNING]
> **開發重要注意事項**：由於遊戲系統頻道的訊息長度較長，當系統發布馬匪刷新通知時，若文字長度超出視窗寬度，遊戲會進行**自動折行 (Line Wrap)**。
> 這會導致「速往」或「前往」等關鍵字被截斷在兩行之間（例如第一行結尾是「前」，第二行開頭是「往」），或者通知內容被分割成多行。
>
> 為解決折行導致關鍵字比對失效的問題，開發時**必須**實作以下資料清洗步驟：
> 1. **文字塊拼接**：將系統頻道監測區塊 OCR 辨識出的多行文字（含有 `\n`）讀入後，**先移除所有的換行符號 `\n` 與空白字元**，將整個區塊的文字融合成一個**連續無換行的單一字串**。
> 2. **全局關鍵字檢索**：在此拼接後的完整字串中，檢索是否存在「前往」或「速往」等關鍵字。這樣即使關鍵字在畫面上剛好換行，合併後也能被順利偵測。
> 3. **時間戳記溯源**：一旦在合併字串中偵測到關鍵字，利用正規表示式往回（向左）尋找**最鄰近且符合 `[HH:MM:SS]` 格式的時間戳記**。這即為馬匪刷新的真正系統時間。

* **判定邏輯**：
  * **去重機制**：若提取出的馬匪刷新時間戳記已存在於該影片已記錄的「馬匪時間戳記集合」中，則忽略（防止同一個影格或連續影格內重複偵測同一條訊息）。
  * 若為新時間戳記，將馬匪次數累加 1。
  * 讀取同影格中「剩餘人數 ROI」所辨識出的數字，作為「此時玩家人數」。
  * **遊戲開始時間**：在合併後的系統頻道文字中，尋找關鍵字 **「开启倒计时:0」**。一旦偵測到，向左溯源最近的 `[HH:MM:SS]` 作為「遊戲開始時間」。若整場對局未偵測到此訊息，則標記為 `"N/A (中途開始)"`。

### 5.3 對局情境判定狀態機 (Match Situation State Machine)
* **對局開始與中途錄製判定 (Segmented Recording)**：
  - 若偵測到的首個圈數即為 $X > 0$，代表影片為中途錄製。此時系統自動將「遊戲開始時間」寫入 `"N/A (中途開始)"`，並在縮圈統計表中，將比 $X$ 還小的圈數（如 0 到 $X-1$ 圈）之時間欄保留為**空白 (Null)**，直接從 $X$ 圈開始記錄。
  
#### 5.3.1 「第 0 圈 (0/8)」之備戰與對局狀態判定
* **問題背景**：當影片一開始的畫面顯示為 `已刷圈/总圈数: 0/8` 時，對局可能處於「備戰等待階段（尚未開始）」或「對局已正式開始（但尚未縮第一圈）」。為精確紀錄「遊戲開始時間」，開發者**必須**實作以下邏輯進行區分：
* **狀態判定條件**：
  1. **判定為「備戰等待中」**：
     - 若在後續的影片畫面上成功偵測到系統頻道出現關鍵字 `开启倒计时:0`（或 `開啟倒計時:0`），則確定先前為備戰狀態。
     - **處理動作**：將該關鍵字行首的 `[HH:MM:SS]` 時間戳記，記錄為**精確的「遊戲開始時間」**。
  2. **判定為「中途加入（且已在第 0 圈中）」**：
     - 若影片播放超過 30 秒，系統頻道中**未**出現 `开启倒计时:0`，但偵測到以下任一事件：
       - **指標 A (擊殺廣播出現)**：系統頻道開始滾動擊殺/重傷播報（如：`[HH:MM:SS] [玩家A] 被 [玩家B] 重傷...`）。
       - **指標 B (人數變動)**：畫面右側的剩餘人數開始減少（如低於 100 人）。
       - **指標 C (縮圈倒數啟動)**：右下角 UI 出現「距離安全區刷新: XX」且秒數每秒遞減。
     - **處理動作**：判定為對局已中途開始，將「遊戲開始時間」標記為 `"N/A (中途開始，已於第 0 圈)"`，以防誤將影片播放起點的時間當作遊戲開局時間。
* **玩家重傷/提早退場判定 (Early Death / Match Termination)**：
  - **偵測機制**：當玩家於對局中死亡、退出遊戲、或是影片進入載入/排隊大廳畫面時，畫面上的吃雞專屬 UI（特別是右下角的「已刷圈/总圈数」以及右側的「剩余人数」區塊）會徹底消失。
  - **判定邏輯**：當程式**連續 5 個抽樣影格 (約 10 秒)** 皆無法從 ROI 中辨識出任何符合「X/8 圈數」或「剩餘人數」的特徵時，狀態機即判定當前對局**已提早終止 (玩家死亡或退場)**。
  - **資料處理**：當前對局停止採樣。未經歷的後續圈數（例如若在第 3 圈退場，則第 4 到 8 圈）在 Excel 報表中一律保留**空白 (Null)**，禁止隨意填充或繼承前值。
* **單一影片多場對局判定 (Multi-match in Single Video)**：
  - 由於直播存檔影片長度可能長達數小時，同一部影片可能包含多場對局。
  - **重啟判定**：若狀態機已判定「對局終止」後，程式在後續影格中**重新偵測到**戰場 UI（如圈數 `X/8` 或剩餘人數）再次出現，即視為玩家開啟了**一場全新的對局**。
  - **狀態重置**：狀態機必須重置所有對局計數器（馬匪次數歸零、清空上一場的馬匪時間戳記集合、將上一場的圈數變數 `current_circle` 設回 `-1`），並在 Excel 報表中**新增一行全新的資料列 (Row)** 進行記錄，避免跨對局資料產生重疊污染。

## 6. 輸出數據規範與報表格式

分析完成後，系統必須將資料輸出至名為 `analysis_results.xlsx` 的 Excel 活頁簿中，包含以下兩個分頁：

### 分頁 1：縮圈時間統計
* **分頁名稱**：`縮圈時間統計`
* **欄位順序與標題**：
  1. `影片連結` (字串)
  2. `圈數` (整數，0 到 8)
  3. `變成這個圈數的最早時間` (時間格式，HH:MM:SS)

### 分頁 2：馬匪時間統計
* **分頁名稱**：`馬匪時間統計`
* **欄位順序與標題**：
  1. `影片連結` (字串)
  2. `遊戲開始時間` (時間格式，HH:MM:SS 或 "N/A (中途開始)")
  3. `馬匪刷新次數` (整數，1, 2, 3...)
  4. `出現馬匪的最早時間` (時間格式，HH:MM:SS)
  5. `此時玩家人數` (整數，0 到 100)

*註：當有新影片分析完成時，系統必須讀取既有的 Excel 檔案，將新數據追加 (Append) 到各分頁の末尾，並進行主鍵去重後重新存檔。*

## 7. 部署與執行規範

### 7.1 GitHub Actions 環境要求
為實現完全背景執行，GitHub Actions 流程應包含以下步驟：
1. **簽出代碼** (actions/checkout)。
2. **安裝系統依賴**：藉由 `sudo apt-get install` 安裝 `tesseract-ocr` 核心與 `tesseract-ocr-chi-sim`（簡體中文語言包）。
3. **安裝 Python 依賴**：包含 `yt-dlp`, `opencv-python`, `pytesseract`, `pandas`, `openpyxl`。
4. **執行辨識**：執行分析腳本，自動讀取 `videos.txt`、下載、分析、並生成 Excel 報表。
5. **自動提交**：將更新後的 Excel 檔案自動 Commit 並 Git Push 回倉庫中。
